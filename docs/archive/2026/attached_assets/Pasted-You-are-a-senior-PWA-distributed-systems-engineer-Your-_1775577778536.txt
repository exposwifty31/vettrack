SECTION 1: OFFLINE QUEUE

Simulate:

- user goes offline
- performs actions
- actions stored in IndexedDB (Dexie-like queue)

- queue length increases
- no network calls are made
- UI reflects optimistic state

---

SECTION 2: RECONNECT + SYNC

Simulate:

- network restored
- queued actions flushed

ASSERT:

- actions sent in correct order
- queue is cleared
- server state is consistent

---

SECTION 3: CONFLICT RESOLUTION (CRITICAL)

Simulate:

- A offline
- B updates online
- A reconnects with stale version

ASSERT:

- server rejects update (version mismatch)
- client performs rollback
- UI reflects server truth
- conflict is logged

FAIL IF:

- overwrite occurs
- silent failure happens

---

SECTION 4: SERVICE WORKER CACHE

Simulate:

- first load → cache assets
- offline reload

ASSERT:

- app loads without network
- assets served from cache
- no crash

---

SECTION 5: MULTI-CLIENT CONSISTENCY

Simulate:

- two clients updating same resource

ASSERT:

- no inconsistent state
- deterministic final state

---

SECTION 6: CAMERA (QR SCAN)

Simulate:

User clicks "Scan QR"

ASSERT:

1. CAMERA OPEN
- getUserMedia is called
- stream.active === true
- video track exists

2. PERMISSION HANDLING
- permission requested only ONCE
- subsequent scans reuse permission

FAIL if:
- permission requested multiple times

3. STREAM VALIDATION
- track.readyState === "live"

4. SCAN COMPLETE → CAMERA STOP
Simulate scan success

ASSERT:
- all tracks stopped
- readyState === "ended"
- stream.active === false

FAIL if:
- camera remains active

5. MULTIPLE SCANS

Simulate:
- open → scan → close → open again

ASSERT:
- new stream created
- no reuse of old stream
- no memory leaks
