# VetTrack Backup & Disaster Recovery Strategy\n\n## Overview\n\nVetTrack runs mission-critical vet clinic operations. Patient data, treatment records, and billing information must be protected against data loss. This document describes the automated backup and recovery strategy.\n\n## Backup Schedule\n\n### Automated Backups\n\nAll backups are full logical dumps (via `pg_dump`) compressed with gzip:\n\n| Schedule | Frequency | Time (UTC) | Retention |\n|----------|-----------|-----------|----------|\n| **Daily** | Every day | 2:00 AM | 7 days (7 backups) |\n| **Weekly** | Sundays | 3:00 AM | 4 weeks (4 backups) |\n| **Monthly** | 1st of month | 4:00 AM | 12 months (12 backups) |\n\n### Backup Format\n\n- **File naming**: `vettrack-{schedule}-YYYY-MM-DDTHH-MM-SS.sql.gz`\n- **Compression**: gzip (typically 10-20% of original database size)\n- **Storage location**: `/var/backups/vettrack/`\n- **Encryption**: TLS in transit; at-rest encryption depends on storage backend\n\n### Example Backup Files\n\n```\nvettrack-daily-2026-07-14T02-00-00.sql.gz   (245 MB, created 2 AM UTC)\nvettrack-weekly-2026-07-13T03-00-00.sql.gz  (250 MB, Sunday 3 AM UTC)\nvettrack-monthly-2026-07-01T04-00-00.sql.gz (260 MB, 1st of month 4 AM UTC)\n```\n\n## Backup Monitoring\n\n### Health Checks\n\nEvery 6 hours, the system verifies:\n- ✅ At least one successful backup exists\n- ✅ Most recent backup is < 25 hours old\n- ✅ Backup files are intact and restorable\n\nIf checks fail:\n- Alert metric `backup_health_failed` is incremented\n- Error logged: `[backup] health check FAILED — no recent backup`\n- Immediate action required\n\n### Admin Endpoints\n\n#### GET /api/admin/backups/list\n\nList all available backups with metadata.\n\n```bash\ncurl -H \"Authorization: Bearer <admin-token>\" \\\n  https://your-domain/api/admin/backups/list\n```\n\nResponse:\n```json\n{\n  \"backups\": [\n    {\n      \"id\": \"backup-1721008800000-abc1234\",\n      \"filename\": \"Authorization: Bearer <admin-token>\" \\\n  -H \"Content-Type: application/json\" \\\n  -d '{\"schedule\": \"daily\"}' \\\n  https://your-domain/api/admin/backups/create\n```\n\nResponse:\n```json\n{\n  \"success\": true,\n  \"backup\": {\n    \"id\": \"backup-1721008800000-abc1234\",\n    \"filename\": hour**\n\nTime from decision to restore until database is back online and verified.\n\n### Recovery Point Objective (RPO)\n\n**RPO: < 24 hours**\n\nMaximum acceptable data loss. Daily backups are taken at 2 AM UTC.\n\n### Restore Procedure\n\n⚠️ **WARNING**: Restore overwrites all current database data and **cannot be undone**.\n\n#### Step 1: Test Restore Locally (Recommended)\n\n```bash\n# Manually test backup integrity and restore capability\npnpm db:backup-now\n\n# Output:\n# ✅ Backup created: vettrack-daily-2026-07-14T02-00-00.sql.gz\n# ✅ Backup file is valid gzip\n# ✅ Backup contains valid SQL (found 1000+ statements)\n```\n\n#### Step 2: Verify Backup Exists\n\n```bash\ncurl -H \"Authorization: Bearer <admin-token>\" \\\n  https://your-domain/api/admin/backups/list\n```\n\nConfirm the backup you want to restore is in the list and `restorable: true`.\n\n#### Step 3: Initiate Restore\n\n```bash\n# IMPORTANT: This requires explicit confirmation header\ncurl -X POST -H \"Authorization: Bearer <admin-token>\" \\\n  -H \"Content-Type: application/json\" \\\n  -H \"X-Confirm-Restore: RESTORE_DATABASE\" \\\n  -d '{\"filename\": \"vettrack-daily-2026-07-14T02-00-00.sql.gz\"}' \\\n  https://your-domain/api/admin/backups/restore\n```\n\nResponse:\n```json\n{\n  \"success\": true,\n  \"message\": \"Database restored from vettrack-daily-2026-07-14T02-00-00.sql.gz. All data has been replaced with backup contents.\"\n}\n```\n\n#### Step 4: Verify Restore\n\n```bash\n# Check application health\ncurl https://your-domain/api/health\n\n# Verify data integrity (app-specific checks)\n# - Patient count matches expectations\n# - Recent appointments are visible\n# - Billing records are consistent\n```\n\n## Testing & Verification\n\n### Weekly Backup Verification\n\nRun this weekly to ensure backups are healthy:\n\n```bash\npnpm db:backup-now\n```\n\nThis script:\n1. Creates a backup\n2. Verifies the backup file is valid gzip\n3. Extracts and validates SQL structure\n4. Reports backup statistics\n5. Does **NOT** restore (non-destructive test)\n\n### Monthly Full Restore Test\n\nOnce per month, in a staging environment:\n\n1. Clone production database to staging\n2. Take a backup of staging\n3. Restore that backup to staging (non-destructive smoke test)\n4. Verify staging application works\n5. Compare staging data to production (spot check)\n\nThis validates:\n- Backup files are actually restorable\n- Restore procedure works as documented\n- Team understands the restore process\n- RTO estimates are realistic\n\n## Troubleshooting\n\n### Backup Failed\n\n**Symptom**: `[backup] health check FAILED — no recent backup`\n\n**Diagnostics**:\n```bash\n# Check backup directory exists and is writable\nls -la /var/backups/vettrack/\n\n# Check disk space\ndf -h /var/backups/\n\n# Check recent backup attempt logs\npnpm db:backup-now\n\n# Check endpoint\ncurl -H \"Authorization: Bearer <token>\" \\\n  https://your-domain/api/admin/backups/stats\n```\n\n**Common Causes**:\n- Disk full (`df -h` shows > 95%)\n- Database connection issues (verify `DATABASE_URL` is set)\n- Permission issues on `/var/backups/vettrack/` directory\n- Database lock or contention (check active queries)\n\n### Restore Failed\n\n**Symptom**: Restore endpoint returns error\n\n**Steps**:\n1. Verify backup file exists: `ls -la /var/backups/vettrack/filename`\n2. Test restore manually:\n   ```bash\n   gunzip < /var/backups/vettrack/filename.sql.gz | head -50 | grep -c CREATE\n   ```\n   Should output a count > 0\n3. Check database connectivity\n4. Review Postgres logs for errors\n5. Ensure application is stopped or connections are drained before restore\n\n### Backup Slow\n\n**Symptom**: Backup takes > 10 minutes\n\n**Cause**: Large table scans or high disk I/O\n\n**Solutions**:\n- Run during low-traffic hours (currently 2 AM UTC)\n- Add indexes on frequently scanned columns (see `pnpm db:analyze-indexes`)\n- Increase backup window (adjust `BACKUP_HOURS` in backup-scheduler.ts)\n- Consider incremental backups (WAL archiving) for faster backups\n\n## Configuration\n\n### Environment Variables\n\n```bash\n# Disable backup scheduler (for testing)\nBACKUPS_DISABLED=true\n\n# Custom backup directory (default: /var/backups/vettrack)\nBACKUP_DIR=/mnt/nfs/backups\n\n# Database connection (required)\nDATABASE_URL=postgresql://...\n```\n\n### Retention Policy\n\nEdit `server/lib/backup.ts` to change retention:\n\n```typescript\nconst BACKUP_RETENTION = {\n  daily: 7,    // Keep 7 daily backups\n  weekly: 4,   // Keep 4 weekly backups\n  monthly: 12, // Keep 12 monthly backups\n};\n```\n\n### Backup Schedule\n\nEdit `server/lib/backup-scheduler.ts` to change
