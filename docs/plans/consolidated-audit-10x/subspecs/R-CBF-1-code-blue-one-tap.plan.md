## Frozen doctrine (every card obeys — non-negotiable)

No new transport (SSE only) · **no offline queueing** (emergency mutations fail loud via `classifyEmergencyEndpoint`) · **server-confirmed end** (never optimistic local termination) · **no emergency endpoint in any cache** · **bounded-enum telemetry only**.

## Design (from the mobile/HIG lens): arm → hold-to-confirm

"One tap" literally = an accidental-Code-Blue generator (phone in a scrub pocket) and fights Apple's deliberate-confirmation rule. Resolution (Apple Emergency-SOS precedent): **tap = arm** (navigate to a full-screen armed screen, unswipeable) → **commit = an exactly-800ms press-and-hold** with an escalating haptic ramp + a filling ring, always-visible **Cancel**. Reads as one gesture, instant under stress, pocket-proof.

### R-CBF-1.1 · Orchestration endpoint (compose, don't rebuild)

- **Goal:** one server action that composes, in order: **(1) claim/look up the idempotency record for `(clinicId, token)` — the FIRST transactional step, before any cart lookup or reservation; on an existing token the response is claim-state-dependent (see Transaction boundary): REPLAY the stored session + the **current durable paging state** with NO cart/session side effects ONLY for a `committed` claim, a **retryable conflict** (no side effects) for a still-active lease, RECLAIM for an `expired`/`aborted` claim** → (2) resolve the **nearest ready** cart → (3) **CAS soft-reserve** it → (4) create the session (`code-blue.ts`) → (5) insert the realtime outbox event → enqueue the team page. **`clinicId`-scoped on every step.**
- **Cart-selection rule (pinned):** "nearest ready" = the ready cart with the smallest **room-adjacency distance** to the **initiating location**, whose source is pinned (no implicit GPS / last-scanned drift): **the initiating user's current room from their open `vt_clinical_check_ins` row → else their last equipment-scan room (last-known) → else, if neither is known, NO distance is used and selection falls back to any ready cart**. **SERVER-AUTHORITATIVE (pinned):** the server **re-reads the authenticated user's clinic-filtered check-in / last-scan location inside the transaction and uses THAT to select the cart** — a client-submitted location is at most an optimistic **hint**, re-validated against the server source, **never trusted directly** (a stale or tampered client value must not steer emergency-cart reservation). The RED asserts a deterministic cart from the **server-derived** location and that a tampered client hint does not change the selection. Adjacency distance uses R-M1.2's model when present, else last-known-location distance; deterministic tie-break by ascending cart id.
- **Idempotency (pinned — request-scoped token, NOT a time bucket):** the client generates **one token per hold gesture** (R-CBF-1.3), persisted across retries; the endpoint enforces uniqueness per **`(clinicId, token)`** and **resolves a duplicate by claim state** (REPLAY a `committed` claim / retryable **conflict** for a still-active lease / RECLAIM an `expired`/`aborted` claim — see Transaction boundary). (A time bucket could merge two legitimate incidents by the same user, or duplicate across a bucket boundary.)
- **Transaction boundary (pinned):** the **idempotency claim is the first step**; then cart lookup + **CAS reservation + session creation + outbox insert commit atomically** (one DB transaction). **The idempotency record is bound to the COMMITTED session — a duplicate replays ONLY when a committed session exists for `(clinicId, token)`.** **Claim lifecycle (pinned — the claim is a SEPARATE durable record, NOT inside the session transaction):** step 1 writes/reads the `(clinicId, token)` claim in its OWN durable write with states `claimed(fence, leaseUntil) → committed → released`; the atomic session transaction (step 2) does reservation + session + outbox insert **and flips the claim to `committed`** in the same commit. Because the claim was written in step 1 (not the session txn), a **session-txn abort leaves the claim in `claimed`** (nothing about the session persists) and its lease lets a later retry reclaim it — this resolves the apparent atomicity contradiction. **Fencing (pinned — a short TTL alone is unsafe against a slow-but-still-active owner):** the claim carries a **monotonic fencing/version token**; reclamation issues a NEW fence, and the create path is guarded so **only the current fence-holder can flip the claim to `committed`** — an owner whose fence was superseded (its lease taken over) **is rejected on commit**, so a retry can never create a second session that then races the original's commit. **Retry response by claim state (pinned):** (1) **`committed`** → **REPLAY** the committed session + the **current durable paging state**, aborting before any cart/session side effect (a losing duplicate never re-reserves). (2) **`claimed`, lease NOT expired, no committed session** → **ACTIVE-LEASE: return a retryable conflict (or a bounded wait for the owner to commit or its lease to expire) with NO cart lookup, NO reservation, NO session, and NO outbox side effect** — a still-active lease is **never** reclaimed and **never** replayed (the in-flight owner is presumed still working; fencing already prevents a double-commit, but a retry must not steal a live lease). (3) **`expired`/`aborted`** (lease elapsed, or the owner released the claim after a session-txn abort — claim present, no committed session) → **RECLAIM: take a fresh fence, RE-CLAIM, and proceed with a new reservation/session transaction; it must NOT replay an empty result nor permanently reject the token.** A fence-holder whose lease was superseded is **rejected on commit** and never replays. **REPLAY only for `committed`; RECLAIM only for `expired`/`aborted`; an active lease is a retryable conflict — never a reclaim, never a replay.** **Paging status is durable AND distinct from the replay result (pinned):** the transaction can only record that paging was **enqueued** — it persists a paging state of `queued | processing | sent | failed`, and a replay returns the **current** state (never a static "success", so a pending job is not reported delivered and a DLQ'd job is not replayed as sent). **Paging rides the EXISTING outbox:** it is written to `vt_event_outbox` within the same commit and delivered by the existing **`startEventOutboxPublisher` (the sole outbox reader)** through its typed event/consumer path with the existing retry + DLQ semantics — **no second outbox polling/drain loop is introduced.** **Never delete a committed session** after its outbox event published — end is server-confirmed only.
- **Compensation:** if the **session** transaction aborts, its writes (CAS reservation + session + outbox insert) roll back together and **nothing about the session persists** — but the durable idempotency **claim** (written in step 1, its own prior durable write) **remains `claimed`** and becomes reclaimable only once its lease expires or the owner releases it (see Claim lifecycle / Retry response by claim state); if the post-commit page job exhausts retries it lands in the DLQ without affecting the session.
- **RED:** `tests/code-blue-one-tap-orchestration.test.ts` — one call yields a reserved nearest-ready cart + a session linked to it + an outbox event + an enqueued page; a **duplicate token for a COMMITTED claim replays the same session with NO second cart reservation attempted**; an **ACTIVE-LEASE retry (claim `claimed`, lease NOT expired, no committed session) returns a retryable conflict with NO cart lookup / reservation / session / outbox side effect (neither replay nor reclaim)**; **two concurrent starts with the same token produce exactly ONE committed session — the second observes the active claim and gets a retryable conflict (no second reservation), and once committed a later retry replays**; **an aborted session transaction leaves no partial session/reservation while the durable claim remains `claimed`**; **an EXPIRED/aborted claim (lease elapsed or released, no committed session) followed by a retry with the same token RE-CLAIMS and creates a fresh committed session + reservation (no empty replay, no permanent token rejection)**; **a duplicate-token replay reports the CURRENT durable paging state (`queued`/`processing`/`sent`/`failed`), and an exhausted-retry `failed` is reported WITHOUT deleting the committed session**; cross-clinic isolation asserted.
- **Guardrail:** all steps online-only; **the endpoint is registered with `classifyEmergencyEndpoint` (`src/lib/offline-emergency-block.ts`) and all client calls go through a typed `src/lib/api.ts` wrapper **with a matching request/response type in `src/types/` (or `shared/`), enforced by `pnpm typecheck`** — no raw fetch may bypass the guard**; no optimistic client state. A focused offline test (see R-CBF-1.5) proves the mutation fails loudly + increments `offline_emergency_mutation_blocked_*` and is never queued.
- **Verify:** `pnpm test -- tests/code-blue-one-tap-orchestration.test.ts` + **`npx tsc --noEmit`** + **`pnpm typecheck`** (also covers the server tsconfig) + `pnpm test` + the Code Blue Playwright drill.

### R-CBF-1.2 · Soft-reserve = additive custody hint (compare-and-set)

- **Goal:** a nullable `reservedForSessionId` hint on cart state — **never blocks** a clinician grabbing a different cart. Set via **compare-and-set: write only where `reservedForSessionId IS NULL`.**
- **Collision rule (pinned):** on a CAS miss (another session reserved it first) the loser **re-resolves to the next eligible ready cart** and never overwrites; if none remain it returns an explicit no-cart-available signal (the session still starts — the reservation is advisory).
- **Cleanup:** a failed **or** ended session clears **only its own** `reservedForSessionId` (scoped by session id), never another session's.
- **RED:** `tests/code-blue-soft-reserve.test.ts` — reserving sets the hint; a checkout of a *different* cart is unaffected; **two concurrent starts: the loser cannot overwrite the winner's `reservedForSessionId`, receives the next cart (or an explicit conflict), and neither clears the other's reservation**; a failed session and an ended session each clear only their own hint.
- **Guardrail:** additive column; no change to custody-toggle semantics; RFID/custody non-goals preserved.
- **Verify:** `pnpm test -- tests/code-blue-soft-reserve.test.ts` + **`pnpm typecheck`** (also covers the server tsconfig) + the Code Blue Playwright drill.

### R-CBF-1.3 · Client arm→hold-to-confirm (the safe "one tap")

- **Goal:** tab-bar emergency slot / `HomeChrome` banner → **full-screen armed screen** (reuse checklist-gated `code-blue.tsx`) → **hold-to-start** control (**exactly 800ms**, `haptics.warning()`→`haptics.locked()` ramp, filling ring) that **generates the per-gesture idempotency token** and may send a location **hint**, but **the server re-derives the initiating location authoritatively** (check-in → last-scan → none, per R-CBF-1.1) — the client value never directly steers cart selection (R-CBF-1.1) → fires R-CBF-1.1 on commit; always-visible Cancel (never traps — ties to the R-CB-01 fix). Phone *initiates*; **iPad and board are display-only — they render server-confirmed sessions and CANNOT arm, hold, or start a session locally**.
- **RED (a11y + hold-boundary are executable, not prose):** `tests/code-blue-hold-to-confirm.test.tsx` — assert: a single tap does NOT create a session; a **completed** hold does (one token); **early release before 800ms does NOT fire (fake-timer boundary: 799ms → no fire, 800ms → fire)**; Cancel dismisses without starting; the control is operable by **keyboard/switch activation** (not pointer-only); **focus enters the armed screen on open and returns to the trigger on cancel/close**; live-log `aria-live` announcements are **throttled/batched** (not one-per-entry); ≥56px targets; reduced-motion fallback for the ring.
- **Guardrail:** every a11y assertion above is test-enforced (emergency flow — no gesture-only or hover-only affordance).
- **Verify:** `pnpm test -- tests/code-blue-hold-to-confirm.test.tsx` + **`pnpm typecheck`** (also covers the server tsconfig) + the Code Blue Playwright drill.

### R-CBF-1.4 · Inline drug-dose reference in the timed log

- **Goal:** a **static, versioned, clinician-approved** drug-dose reference shown inline in the timed-log view (reference data, not a new domain). **Clinical provenance is MANDATORY — rendering alone is insufficient (this is clinical-critical):** each entry carries a **named clinician-approved source**, a **version + effective date**, explicit **species / weight-band / concentration / unit** scope, and a **named review/update owner**; the bundled table is versioned and its provenance is shown with the reference.
- **RED:** `tests/code-blue-drug-reference.test.tsx` — the reference renders with its **source + version/effective-date + species/weight/concentration/unit scope + review owner**; an entry **missing any provenance field FAILS validation** — **and so does an entry with an EMPTY/placeholder source or owner, a malformed version or effective-date, or an out-of-scope/unsupported species/weight/concentration/unit** (presence alone is insufficient; unsafe dose guidance must not satisfy a mere non-null check); he+en; no PII; no network dependency.
- **Guardrail (clinical safety):** stale or unsourced dose guidance is a safety defect — provenance fields are enforced, not optional; clinic-configurability deferred beyond v1.
- **Verify:** `pnpm test -- tests/code-blue-drug-reference.test.tsx` + **`pnpm typecheck`** (also covers the server tsconfig) + the Code Blue Playwright drill.

### R-CBF-1.5 · e2e drill + doctrine verification (acceptance bar)

- **Playwright drill** (Phase-9 style): one arm→hold → cart reserved + **team page ENQUEUED (paging state observed as `queued` or `processing` at minimum — NOT asserted as `sent`, since delivery is async via `startEventOutboxPublisher`)** + log open + **board propagation**, observed live.
- **Durable paging state (acceptance bar, explicit):** the paging states are `queued | processing | sent | failed`; a successful arm→hold must reach **at least `queued`/`processing`** (delivery is async — never assert `sent` synchronously); a duplicate-token replay reports the **CURRENT** state (`queued`/`processing`/`sent`/`failed`), not a static "success"; an exhausted-retry lands `failed` in the DLQ **without deleting the committed session** (asserted in `tests/code-blue-one-tap-orchestration.test.ts`).
- **Offline block:** an offline attempt blocks loudly and increments `offline_emergency_mutation_blocked_*`.
- **Server-confirmed end:** session end still follows the SSE event (no optimistic local termination).
- **Verify:** the Code Blue Playwright drills + **`npx tsc --noEmit`** + **`pnpm typecheck`** (also covers the server tsconfig) + `pnpm test`.
