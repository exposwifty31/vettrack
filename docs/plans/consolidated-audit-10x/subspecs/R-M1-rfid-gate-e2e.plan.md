### R-M1.1 · Managed reader entity (replaces script/manual flow)

Promote "reader" from inferred to a first-class managed entity with CRUD + provisioning + health.

- **a) schema (directional + tenant-safe at the DB boundary):** `vt_rfid_readers` (`clinicId, id, name, gatewayCode, roomId FK, fromRoomId FK, toRoomId FK, **gateType (`internal | boundary | dock`)**, physicalLocation, status, lastSeenAt, **lastReaderHeartbeatAt** (dedicated reader-health timestamp — see R-M1.1d), provisioningState, createdAt`) — **`gateType` is the typed, deterministic boundary classification** the egress rule (R-M1.2c) keys on; `physicalLocation` is never used for routing. **Gateway ownership (pinned — single source of truth):** `vt_rfid_readers.gatewayCode` is the **authoritative** gateway↔reader registry; the legacy `vt_rooms.gatewayCode` that drives ingest today is **migrated into `vt_rfid_readers` (one-time backfill: each existing `rooms.gatewayCode` → a managed reader with `roomId` = that room and `provisioningState='legacy_unconfigured'` — `gateType`/adjacency left UNSET, so it serves last-seen but NOT directional egress until an admin completes configuration), after which ingest resolves gateways through `vt_rfid_readers` only** — the two never coexist as independent mappings. **`roomId` vs `fromRoomId`/`toRoomId`:** `roomId` is the reader's **canonical physical mounting room** (where the device sits); `fromRoomId`/`toRoomId` are the **directional adjacency endpoints** for gate semantics — for an `internal` gate `roomId` is one of the two, for a `boundary`/`dock` gate `roomId` = the single internal endpoint (the external side is the `NULL` zone). A migration/CRUD validation **rejects a reader whose `roomId` is neither `fromRoomId` nor `toRoomId`**, so the registry and room mapping cannot drift. **Tenant-safe constraints (pinned — enforced in the DB, not just service queries):** **composite unique `(clinicId, gatewayCode)`** (not a bare "unique per clinic"); `roomId`/`fromRoomId`/`toRoomId` must reference rooms in the **same `clinicId`** — enforce via **composite foreign keys** (`(clinicId, roomId)`, `(clinicId, fromRoomId)`, `(clinicId, toRoomId)` each → `vt_rooms(clinicId, id)`; add `UNIQUE (clinicId, id)` on `vt_rooms` if absent), **not a CHECK** (a CHECK can't enforce cross-row referential integrity). **Directional-pair validity is `gateType`-scoped (pinned):** an **`internal`** gate has **both** `fromRoomId` + `toRoomId` non-null, same-clinic, and **distinct**; a **`boundary`/`dock`** gate has **exactly one** non-null internal room endpoint, and the other endpoint is the **external zone represented as `NULL`** — an explicit out-of-clinic convention, **never a placeholder/fake `vt_rooms` row**; the composite FKs are **nullable** and still enforce same-clinic on every non-null endpoint. **Reject** a fully-empty pair, a self-referential pair (`fromRoomId = toRoomId`), an `internal` gate with any null endpoint, and a `boundary`/`dock` gate with two non-null or two null endpoints. **(A `provisioningState='legacy_unconfigured'` reader with unset `gateType` is EXEMPT from these pair rules until configured — it has no directional role yet.)** Migrate; `npx drizzle-kit generate` → commit SQL. **RED:** `tests/migrations/rfid-readers.test.ts` (DB-integration) — directional columns exist; **a duplicate `(clinicId, gatewayCode)` is rejected; a cross-clinic `roomId`/`fromRoomId`/`toRoomId` is rejected; a self-referential or half-populated directional pair is rejected; a reader whose `roomId` is neither `fromRoomId` nor `toRoomId` is rejected; ingest resolves gateways via `vt_rfid_readers` (not the legacy `rooms.gatewayCode`)**.
- **b) service + CRUD route** — extend `server/services/rfid-readers.service.ts` from derived-list to entity CRUD (create/rename/deactivate); new mutation endpoints behind `requireAdmin`, registered in `server/app/routes.ts`; join the reader's **own** heartbeat for status — the dedicated **reader-heartbeat / accepted-reader-event timestamp** defined in R-M1.1d, **NOT `equipment.lastRfidGatewayCode/lastRfidSeenAt` asset-read traffic** (a healthy-but-quiet reader with no equipment passing it must not read as offline). **RED:** `tests/rfid-readers-crud.test.ts` (create/rename/deactivate; `clinicId`-scoped; cross-clinic denied).
- **c) self-serve provisioning + ingest toggle + HMAC rotation contract** — endpoint to provision/rotate the per-clinic HMAC secret (replaces `scripts/rfid/provision-secret.ts`) + toggle `rfid.ingest_enabled.<clinicId>` from the UI; reuse `getCredentials`/credential-manager + `server/lib/rfid/config.ts`. **Security (pinned):** `requireAdmin`; `clinicId` derived **from the authenticated context only** (never request input); the secret is returned **once** in the provision response and is **never logged or cached** (no response caching). **Rotation contract (pinned — retry-safe + concurrency-safe):** a client-supplied **rotation idempotency key** with persisted state — a retry with the same key **returns the original result envelope** (**the persisted record stores the rotation *status*, NOT the plaintext secret**), issuing no second secret and invalidating none. **The secret is delivered exactly once, in the first successful response; if that response is lost, recovery is a NEW rotation (new key)** — idempotency prevents double-rotation, it does not re-deliver a lost secret. Also: **at most one in-flight rotation (version) per clinic** so two admins can't overwrite `current`/`previous`; rotation **snapshots the eligible active readers at start** and requires acknowledgement only from that snapshot (readers added mid-grace don't block; a clinic with **no active readers** completes immediately). Ingest verifies **current OR previous** during the grace window (`rotationStartedAt` → `+graceTTL`); on grace expiry **or** all-snapshot-readers-acknowledged the **previous is invalidated** — and **rollback becomes unavailable at that same instant**. **Rollback is valid ONLY while `previous` is still retained** (before grace expiry AND before the acknowledgement-driven invalidation, whichever comes first): a rollback in that window restores `previous` as `current` **and invalidates the newly issued secret**; **once `previous` is invalidated there is nothing to roll back to — recovery is a fresh rotation** (a new key). This removes the contradiction where early acknowledgement could invalidate `previous` before the nominal expiry yet still appear rollback-eligible. **RED:** `tests/rfid-provisioning.test.ts` — admin-only; `clinicId` from auth; secret returned once + not logged/cached; **a same-key retry returns the original (no double-rotation); two concurrent rotations → one wins, the other is rejected (no overwrite); during grace either secret verifies; after grace/ack the previous is rejected; rollback restores the previous + invalidates the new secret**; cross-clinic denied.
- **d) reader-offline detection** — a scheduler/job computing staleness. **Pinned semantics:** a **per-clinic staleness threshold** on the reader's **own heartbeat** — the concrete **`vt_rfid_readers.lastReaderHeartbeatAt` column** (distinct from `lastSeenAt`), **server-set on every accepted reader-level event — a reader heartbeat ping OR an accepted ingest batch from that reader** (never a client-supplied timestamp), **NOT `equipment.lastRfid*` asset-read traffic** (a healthy-but-quiet reader with no equipment passing it must NOT be marked offline); no update to `lastReaderHeartbeatAt` within the threshold → offline; a fixed **scheduler cadence**; explicit **`healthy→offline` and `offline→healthy` transitions**; the `rfid_reader_offline` signal (feeds R-M1.3) is emitted **only on a status *change*** — while a reader stays offline (or stays healthy) no repeat signal/outbox row is emitted (deduplicated). Register in `server/app/start-schedulers.ts`. **RED:** `tests/rfid-reader-offline.test.ts` — crossing the threshold emits **one** offline signal; **offline→healthy emits one clear; repeated sweeps while unchanged emit nothing (dedup)**.
- **e) admin console CRUD UI** — turn `RfidReadersConsolePage` from read-only into CRUD (add/rename/deactivate, provision secret, ingest toggle, offline badges); reuse the console CRUD pattern from `src/pages/console/WebhooksConsolePage`/`GovernanceConsolePage`; `api.rfidReaders` gains create/update/deactivate/provision (`src/lib/api.ts`) + `src/types/`. Route stays `WebOnlyGuard + ManagementGuard`. **RED:** `tests/rfid-readers-console.test.tsx` (renders CRUD; offline badge; management-gated).
- **Guardrail:** ingest auth unchanged (reuse the webhook HMAC pattern); no custody writes.

### R-M1.2 · True directional gates (entry/exit + adjacency)

- **a) ingest schema (deterministic + tenant-safe):** extend `RfidBatchSchema` (`server/routes/rfid.ts:18-31`) with `direction` (`entered`|`exited`) and/or `fromGateway`/`toGateway`. **Precedence (pinned):** `entered`/`exited` is interpreted **relative to the reader's configured `fromRoomId`/`toRoomId`**; when both `direction` and `fromGateway`/`toGateway` are supplied they **must agree** (else reject); gateway ids resolve **only within the authenticated clinic** — an **unknown or cross-clinic gateway is rejected** (never inferred). **Legacy last-seen fallback applies ONLY to payloads with no directional data at all** — a partial or contradictory directional payload is a **4xx**, not a silent downgrade. **RED:** `tests/rfid-ingest-direction.test.ts` — a valid directional payload persists; a legacy (no-direction) payload still works; **a partial pair, a direction/gateway disagreement, and an unknown/cross-clinic gateway are each rejected (not silently downgraded)**.
- **b) adjacency model** — model room/gate adjacency (which rooms a gate connects). Minimal: a gate (reader) has `fromRoomId`/`toRoomId`; `vt_equipment_rfid_reads` already has `fromRoomId`/`toRoomId` (`:164-186`) — populate them from direction instead of leaving null. **RED:** `tests/rfid-adjacency.test.ts` (directional read writes from/to rooms).
- **c) resolver uses direction** — "exited ER → entered Ward" produces last-seen = destination room. **External destination (pinned):** for a `boundary`/`dock` gate, an exit toward the **external-zone (`NULL`) endpoint** resolves destination = **external zone** (not a room, and not an "unknown-room" error) — which feeds the egress case below. **Egress signal (pinned, idempotent):** an exit through a gate whose **`gateType ∈ {boundary, dock}`** (the typed reader field, R-M1.1a — never inferred from `physicalLocation`) with **no matching entry** emits a bounded-enum **`possible_egress`** signal surfaced by R-M1.3 (NOT a separate alert, NOT deferred). **Correlation (pinned):** correlation key = `(clinicId, equipmentId, gateId, **sourceEventId**)` — the source event/batch id (or a deterministic event fingerprint) is **part of the key**, so two legitimate egresses through the same gate in the window are distinct from a retry of one event; matching uses a **bounded ordering/time window**; **retries and out-of-order RFID batches are deduplicated** against persisted correlation state, so exactly one signal fires per genuine egress. **RED:** `tests/rfid-direction-resolve.test.ts` — directional resolve to the destination room; a boundary-exit-without-entry emits **exactly one** `possible_egress`; **a retry emits no duplicate; a reordered/out-of-order batch produces no duplicate or contradictory signal**.
- **Guardrail:** additive to the existing reads table; legacy non-directional ingest must remain valid; no custody mutation.
